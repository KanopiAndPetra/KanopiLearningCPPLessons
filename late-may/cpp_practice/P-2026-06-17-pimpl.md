# C++ practice 2026-06-17 — PIMPL (Pointer to Implementation)

## What I set out to learn

The Jun 16 lesson split a `Box` / `Inventory` pair into a
header + `.cpp` shape — the day-one build-system skill.
This lesson takes the next step: **PIMPL**, a refinement of
that shape that hides every private member of a class
behind a pointer to a struct defined in the `.cpp`.

Three things I want this lesson to teach:

1. **The header is the API; the `.cpp` is the implementation;**
   **and PIMPL takes this to its logical extreme.** With a
   non-PIMPL class, every private member still lives in the
   header, so adding or changing a private field still
   changes the class layout and forces every includer to
   recompile. PIMPL hides *all* private state behind a
   `std::unique_ptr<Impl> impl_`, so the class layout is
   fixed at one pointer regardless of how the private
   state evolves.
2. **The "Big Five" of a PIMPL class is non-obvious.** A
   `unique_ptr` is move-only, so a PIMPL class can't rely
   on the compiler-generated copy ctor. And because the
   dtor must call `delete impl_`, and `delete impl_`
   needs `Impl` to be a complete type, the dtor must be
   defined in the `.cpp`. Move ops can be `= default`-ed
   in the header because moving a `unique_ptr` is just a
   pointer transfer.
3. **The PIMPL pattern has a famous footgun.** Forgetting
   to declare the dtor in the header (or declaring it
   `= default` in the header with `Impl` still incomplete)
   gives a "member access into incomplete type" error at
   *every* call site where the object is destroyed — and
   the error message is far from the cause. The lesson
   shows the failure mode side-by-side with the canonical
   shape so the difference is obvious.

## Key ideas

### The header is now actually stable

In the Jun 16 lesson, the `box.h` header still listed the
private member `std::string label_`. Adding a `weight_`
field to `Box` would have meant editing the header, which
in turn would have invalidated every translation unit that
included `box.h` — even TUs that don't care about weights.
That's the "header churn" problem that real codebases
battle every day.

PIMPL fixes this by putting *every* private member into a
struct defined in the `.cpp`:

```cpp
// pimpl_box.h
class Box {
public:
    explicit Box(const std::string& label);
    ~Box();
    // ... ctors, assigns, methods ...
private:
    struct Impl;                              // forward decl only
    std::unique_ptr<Impl> impl_;
};
```

```cpp
// pimpl_box.cpp
struct Box::Impl {
    std::string label_;
    std::size_t version_ = 0;
    std::function<std::size_t(const std::string&)> hasher_;
    explicit Impl(const std::string& l) : label_(l), hasher_(std::hash<std::string>{}) {}
};
```

Now `sizeof(Box) == sizeof(std::unique_ptr<Box::Impl>)`
(8 bytes on this machine), and it stays 8 bytes no matter
how many fields you add to `Impl`. The header's view of
`Box` is fixed; only the `.cpp` knows what `Box` actually
contains. Add a 1KB scratch buffer to `Impl` and `Box` is
still 8 bytes. The trace from Section 2 of the program
shows this directly:

```
sizeof(Box) = 8 bytes  (just one unique_ptr<Impl> -- nothing else)
```

### The "Big Five" of a PIMPL class

The tricky part of PIMPL is that the class owns a
`std::unique_ptr`, which is *move-only* and is
*non-trivially destructible* (its dtor needs to see the
pointee's full type). For a class like `Box`, the rules are:

| Special member | Where it goes | Why |
|----------------|---------------|-----|
| Default ctor | `.cpp` (constructs the `Impl`) | The ctor calls `std::make_unique<Impl>(label)`, which needs `Impl` complete. |
| Destructor | Declared in header, defined `= default` in `.cpp` | The header declaration suppresses the implicit move ops (Rule of Five) and tells the compiler "I'll define this later." The `.cpp` definition is where `~Impl()` is finally visible. |
| Copy ctor | Declared in header, defined in `.cpp` | `unique_ptr` is non-copyable. We *declare* a copy ctor so the compiler doesn't try to generate one in the header (where `Impl` is incomplete); we *define* it in the `.cpp` as a deep copy. |
| Copy assign | Same as copy ctor | Same reason. |
| Move ctor | `= default` in the header | Moving a `unique_ptr<Impl>` is fine even when `Impl` is incomplete — the move just transfers the pointer. |
| Move assign | `= default` in the header | Same reason. |

The reason this works is asymmetric: the move ops only
*transfer* the pointer (no `delete`, no dereference of
`Impl`), so they can be defaulted in the header where
`Impl` is incomplete. The dtor and copy ops *use* `Impl`
(deallocate it, copy it), so they need `Impl` to be
complete and must therefore live in the `.cpp`.

If you forget any of this, you get the famous PIMPL
footgun. Section 4 of the program shows the broken shape
(`BrokenPimpl`) — a struct that owns a `unique_ptr<Impl>`
but has no user-declared dtor — and the comment block
explains the error you'd see at every use site:

```cpp
struct BrokenPimpl {
    struct Impl;                          // forward decl only
    std::unique_ptr<Impl> impl_;
    // No ~BrokenPimpl() declared -> the compiler tries to
    // generate one at the first point of use, but ~unique_ptr
    // needs Impl complete. Result: "member access into
    // incomplete type" at every Box destruction point.
};
```

In contrast, the canonical shape has every required
declaration in the header:

```cpp
struct CanonicalPimpl {
    struct Impl;
    std::unique_ptr<Impl> impl_;
    CanonicalPimpl();
    ~CanonicalPimpl();
    CanonicalPimpl(CanonicalPimpl&&) noexcept;
    CanonicalPimpl& operator=(CanonicalPimpl&&) noexcept;
    CanonicalPimpl(const CanonicalPimpl&);
    CanonicalPimpl& operator=(const CanonicalPimpl&);
};
```

Same memory layout. Wildly different compile behaviour.

### The deep copy in PIMPL

A subtle point that's easy to miss: when a PIMPL class
*is* copyable (because we declared and defined a copy
ctor in the `.cpp`), the copy must be a *deep* copy, not
a pointer copy. If the copy ctor just did
`impl_(other.impl_)`, the two objects would share a
single `Impl`, and on destruction both would try to
`delete` the same pointer. Double-free.

The canonical PIMPL copy ctor is:

```cpp
Box::Box(const Box& other)
    : impl_(std::make_unique<Impl>(*other.impl_)) {}   // deep
```

This calls `Impl`'s copy ctor (the compiler-generated one
works because `Impl`'s members are all copyable), giving
each `Box` its own `Impl`. Section 1 of the program
proves this works by copying `c` into `f`, mutating `f`,
and showing that `c` is unaffected. The relevant lines
in the output:

```
f = c (copy): f.label() = red
g = f (copy): g.label() = red
after f.set_label("f-was-changed"):
  f.label() = f-was-changed
  c.label() = red  (c is unchanged -- deep copy)
  g.label() = red  (g is unchanged -- deep copy)
```

If the copy were shallow, `c.label()` would also have
become `"f-was-changed"`. It didn't. The deep copy
worked.

### The ABI / rebuild benefit

The reason this idiom exists in the first place is the
*binary-compatibility* and *rebuild* benefit. In a project
with 200 translation units, the difference between
"adding a private field" and "not changing the header" is
the difference between "rebuild 200 files" and "rebuild
1 file." PIMPL makes private state changes invisible to
the header, which means:

1. **Faster rebuilds.** Touching `Impl` invalidates only
   `pimpl_box.cpp`. TUs that include `pimpl_box.h`
   *don't* see the change because the header didn't
   change.
2. **Stable ABI.** If `Box` is shipped as part of a
   shared library, the layout of `Box` is fixed in the
   header (`unique_ptr<Impl>` is always 8 bytes on
   x86-64). A newer version of the library can grow
   `Impl` arbitrarily and existing client binaries
   still link and work, because the layout they see
   didn't change.
3. **Faster compilation of includers.** A PIMPL header
   pulls in strictly less than a non-PIMPL header. No
   `<string>`, no `<vector>`, no `<map>` (unless the
   *public* surface uses them). Less to parse, less
   to compile, less to chase transitive includes.

These are the same reasons Google, LLVM, and Qt use
PIMPL (or close variants) for many of their public
classes. It's not a free lunch — the cost is one heap
allocation per object and one pointer indirection per
method call — but the trade is usually worth it for
public, long-lived API classes.

### The cost, measured in this program

The `hash()` method in `Box` does this:

```cpp
std::size_t Box::hash() const {
    return impl_->hasher_(impl_->label_) ^ impl_->version_;
}
```

Each call dereferences `impl_` twice and calls through
a `std::function` (which itself is an indirection or
more). That's the PIMPL cost. In a hot loop, the
indirection and the function-pointer call inside
`std::function` would be measurable. For typical use,
it's noise.

In this program `Box` is also doing a `std::function`
construction (the `hasher_` member) at object creation.
That's another hidden cost: each `Box` ctor allocates
memory for the function target. I included it
deliberately to make the "PIMPL hides state, but state
has its own cost" lesson concrete. A real PIMPL
`Impl` would not necessarily carry a `std::function`;
it depends on what the class actually needs.

### The alternative shapes, and when to use them

PIMPL is one of several ways to structure a class. The
others worth knowing about:

- **Header-only.** Everything in the header. Templates
  and `inline` functions force this shape. Build time
  goes up linearly with header size, but the binary is
  smaller (no indirection) and there's nothing to link.
- **Non-PIMPL multi-file.** The Jun 16 shape: header
  has the class declaration with private members; the
  `.cpp` has the definitions. Simple, fast, but every
  private change touches the header.
- **PIMPL.** This lesson. Stable header, deep copy
  boilerplate, one indirection per call, one heap
  allocation per object.
- **Shared library / `dylib`.** A `Box` shipped as a
  `.dylib` with no PIMPL is similar to PIMPL in that
  the layout is hidden, but the *interface* is still
  a non-PIMPL class. Clients compile against a header
  that exposes the full layout; the linker resolves
  symbols at runtime. ABI is stable *only* if the
  header doesn't change.
- **PIMPL with a `shared_ptr`** (sometimes called
  "PIMPL with reference semantics"). Makes the class
  copyable without deep-copy code, at the cost of
  reference counting and an extra heap allocation.
  Use when copy semantics are more important than
  performance.

## Design choices and trade-offs

### Why I kept the PIMPL `Box` separate from the Jun 16 `Box`

The Jun 16 `Box` is a regular non-PIMPL class used by the
multi-file `Inventory`. If I had edited `box.h` /
`box.cpp` to add PIMPL, the Jun 16 example would have
stopped compiling (its `Inventory::boxes_` is a
`std::vector<Box>`; a PIMPL `Box` would change how the
vector stores and destroys elements). Keeping the two
lessons separate (`box.{h,cpp}` and `pimpl_box.{h,cpp}`)
means each lesson's code can be read and run in
isolation. The naming convention `pimpl_*` is the
giveaway that these files belong to a different lesson.

### Why I gave `Impl` a `std::function` member

The `hasher_` member is a deliberate cost. It's there to
make the "PIMPL hides state, but state has its own
overhead" point visible. A real `Box` would probably not
have a function pointer for hashing; it would compute
`std::hash<std::string>{}(label_)` directly. The
`std::function` adds a heap allocation per `Impl` and a
function-pointer call per `hash()`. That's the *visible*
cost. I wanted the lesson to be honest about what PIMPL
costs you.

### Why move ops are `noexcept` but the dtor is not `noexcept`

The dtor of `Box` runs the dtor of `unique_ptr<Impl>`,
which runs the dtor of `Impl`, which destroys the
`std::function` inside `Impl`. None of that is declared
`noexcept`, and `std::function`'s dtor can in principle
throw (it normally doesn't, but the compiler doesn't
know that). So `~Box()` is not implicitly `noexcept`,
and I don't pretend otherwise.

The move ops are `noexcept` because they only transfer
the `unique_ptr`. No `Impl` dtor runs, no `std::function`
destructs, no exception possible. This matters for
`std::vector<Box>` reallocation: with `noexcept` moves,
the vector moves on reallocation (fast); without, it
copies (and triggers the deep-copy code).

### Why I chose `= default` in the header for the move ops

Some PIMPL implementations write out the move ctor and
move assign in the `.cpp`:

```cpp
// pimpl_box.cpp
Box::Box(Box&& other) noexcept : impl_(std::move(other.impl_)) {}
Box& Box::operator=(Box&& other) noexcept {
    impl_ = std::move(other.impl_);
    return *this;
}
```

That works, but it's the same code the compiler would
generate if you wrote `= default` in the header. The
header `= default` is shorter, more obviously correct,
and signals "this is a member-wise move — nothing
clever." I prefer it for the move ops. The dtor and the
copy ops, on the other hand, *must* live in the `.cpp`
because they need `Impl` to be complete.

*PIMPL: the header is one pointer, the `.cpp` is everything
else. The trade is one heap allocation per object and one
indirection per method call, in exchange for a stable header
and a much smaller rebuild graph. The boilerplate looks
weird the first time — five special members, four of them
declared in the header, three of them defined in the `.cpp`

forgetting the dtor declaration in the header; the fix is
to *always* declare the dtor in the header for a PIMPL
class, and let the `.cpp` finish it. After that, the
pattern is just a discipline: header for API, `.cpp` for
state, linker for the join.*
