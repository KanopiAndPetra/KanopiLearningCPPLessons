## Why this topic

Every modern-C++ session so far has used `std::move` — moving `unique_ptr`s on Jun 4, returning `Project` owners on Jun 7, moving into `vector::push_back` on Jun 8. But the *mechanics* of why `std::move` works, what rvalue references really are, and how `std::forward` is different have been treated as a black box. Time to crack it open.

## The one idea

C++ has **value categories** (lvalue, prvalue, xvalue) that determine which overload binds, and **rvalue references** (`T&&`) extend the lifetime of temporaries while also selecting the "this is the cheap-to-move-out-of" overload. `std::move` and `std::forward` are just casts: they don't move anything; they just relabel an expression as an rvalue so the rvalue-reference overload can be picked.

## Key concepts I cemented

### Value categories in 30 seconds

- **lvalue** — has identity, persists beyond the expression. Most named variables. `x`.
- **prvalue (pure rvalue)** — no identity, a temporary. `x + 1`, `42`, `f()` (unless it returns a reference).
- **xvalue (expiring value)** — an object whose resources can be reused. The result of `std::move(x)`. Conceptually "an lvalue we have agreed to treat as disposable."

Binding rules that drive everything else:
- `T&` binds to lvalue
- `const T&` binds to anything (lvalue, const lvalue, rvalue, temporary) — extends the temporary's lifetime
- `T&&` binds to rvalue (prvalue or xvalue) — also extends the temporary's lifetime

### `std::move(x)` is a no-op

```cpp
auto&& r = std::move(a);
std::cout << a.data;  // a is UNCHANGED
```

`std::move` is exactly `static_cast<T&&>(a)`. It only changes the *expression's category* — it does not touch the object, run a constructor, or transfer a buffer. The actual transfer happens when a move constructor or move assignment operator is then selected and runs.

The real cost of a move lives in the move constructor — for `std::string`, that's a pointer swap; for a hand-rolled type, it's whatever you write. Until then, `std::move` is free.

### Reference collapsing (the four cases)

When the type system has to combine a deduced `T` with a `&` or `&&` (this happens inside templates), the rules are:

| T    | Param form | Combined | Result  |
|------|------------|----------|---------|
| `U&` | `T&&`      | `U& &&`  | `U&`    |
| `U`  | `T&&`      | `U&&`    | `U&&`   |
| `U&` | `T&`       | `U& &`   | `U&`    |
| `U`  | `T&`       | `U&`     | `U&`    |

This is the single rule that makes forwarding references work. When you pass an lvalue, `T` deduces to `U&`, the `&&` collapses away, and the parameter type is `U&`. When you pass an rvalue, `T` deduces to `U`, the `&&` stays, and the parameter type is `U&&`. The program in section 6 prints exactly this for three call sites.

### `std::move` vs `std::forward`

- `std::move(x)` unconditionally casts `x` to an rvalue. Use when you *know* you want to enable moving from `x` and `x` is no longer needed in its current form.
- `std::forward<T>(x)` casts `x` to an rvalue *only if `T` was deduced as a non-reference type* — i.e., only if the caller originally passed an rvalue. Use in *templates* to preserve the caller's original value category.
- Functionally, in a template, `std::forward<T>(x)` is the right tool when you want to "pass through" arguments; `std::move(x)` would be wrong because it would force a move even when the caller passed an lvalue.

### `noexcept` on the move constructor matters

A `noexcept` move constructor lets `std::vector::push_back` use moves instead of copies during reallocation. If the move can throw, the vector has to copy as a fallback — and now your "move-only" type forces a reallocation-time copy. Always mark your move constructor and move assignment `noexcept` if they can be.

### Don't `std::move` a local in a return

```cpp
std::string make() {
    std::string s = "x";
    return std::move(s);   // BAD: blocks NRVO
}
```

The compiler is allowed to *elide* the copy/move entirely when you `return local;` (RVO/NRVO). Wrapping the local in `std::move` is an explicit rvalue cast, which is treated as an "escape" and **disables the elision** — now you actually run a move. The clang warning `-Wpessimizing-move` catches this; gcc's `-Wreturn-std-move` does the same.

Exception: when you're returning a *parameter*, you do want `std::move` (or pass-by-value), because there's no local for RVO to elide onto.

## Bugs and gotchas I hit

1. **`is_lvalue_reference_v` / `is_rvalue_reference_v` need `<type_traits>`.** I forgot the include initially; clang flagged both names as missing.
2. **A forwarding-reference helper `describe<T>(identity_t<T>{})` is over-engineered.** I had a `type_identity` trait in there to *force* a function template argument, then immediately wrote a `describe<T>()` that didn't need an argument. Stripped the trait and the helper is much clearer.
3. **`-Wpessimizing-move` is right, but the lesson needs the example.** Section 9 of the program *is* the pessimization. I added a `#pragma clang diagnostic push/ignored/pop` around the one line so the build is clean, with a comment explaining the suppression.
4. **`(void)x + 1` is not a thing.** `(void)x` produces a `void` expression, which you can't add to. Replaced with `[[maybe_unused]] int dummy = x + 1; (void)dummy;`.

## Key takeaways

- **Value categories are the substrate.** Everything else — overload resolution, lifetime extension, perfect forwarding, move elision — is a consequence of the lvalue / prvalue / xvalue distinction.
- **`std::move` and `std::forward` are casts.** They don't move anything; they relabel an expression. The actual move happens when a move ctor/assign is selected.
- **Use `std::move` in three places:** to enable move-from a named local before it's destroyed, to forward a *consumed* parameter in the body of a function, and never in a `return local;` line.
- **Use `std::forward` inside templates** to preserve the caller's value category. `std::move` in a template is almost always wrong.
- **Reference collapsing is the single rule** that makes `T&&` a forwarding reference. Memorize the four lines.
- **Mark moves `noexcept`** so containers can use them during reallocation.
